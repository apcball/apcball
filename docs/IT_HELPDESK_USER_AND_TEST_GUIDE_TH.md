# คู่มือการใช้งานและทดสอบ IT Helpdesk

เอกสารนี้ใช้เป็นคู่มือสำหรับผู้แจ้งปัญหา (User) และเป็นขั้นตอนทดสอบระบบสำหรับทีม IT/QA ของโมดูล `buz_it_helpdesk` บน Odoo 17

> **หมายเหตุ:** ชื่อเมนูและสถานะในเอกสารนี้อ้างอิงระบบปัจจุบัน หากมีการปรับแต่งบนแต่ละบริษัท ให้ยึดชื่อที่แสดงจริงในหน้าจอเป็นหลัก

## 1. บทบาทผู้ใช้งาน

| บทบาท | หน้าที่หลัก |
|---|---|
| Requester / User | แจ้งปัญหา ดู Ticket ของตนเอง ตอบกลับ และแนบไฟล์ |
| Agent | รับงาน ตรวจสอบ แก้ไข ตอบกลับ และเปลี่ยนสถานะ Ticket |
| Manager | ดูแล Ticket ของบริษัท จัดการทีม SLA และข้อมูลตั้งต้น |

ผู้ใช้ Portal ต้อง Login ก่อนใช้งาน และจะเห็นเฉพาะ Ticket ที่ตนเองเป็นผู้แจ้ง

## 2. ช่องทางแจ้งปัญหา

### 2.1 แจ้งผ่าน Portal

1. Login เข้า Odoo
2. เปิด Portal แล้วเลือก **My Helpdesk**
3. กด **Create Ticket**
4. กรอกข้อมูลต่อไปนี้
   - **Subject:** หัวข้อสั้นและชัดเจน เช่น `เปิดไฟล์ Excel ไม่ได้`
   - **Category:** เลือกประเภทปัญหา เช่น Hardware, Software, Network หรือ Access Request
   - **Priority:** เลือกระดับความเร่งด่วนตามหัวข้อ Priority ด้านล่าง
   - **Description:** ระบุอาการ ขั้นตอนที่ทำก่อนเกิดปัญหา และผลที่พบ
5. แนบภาพหน้าจอหรือไฟล์ที่เกี่ยวข้องได้หลายรายการ
6. กด **Submit Ticket**
7. จดเลข Ticket ที่ระบบสร้าง และใช้เลขนี้ในการติดตามงาน

### 2.2 แจ้งผ่าน Email

1. ส่ง Email ไปยัง Email Alias ของ Helpdesk ที่บริษัทกำหนด
2. ใช้ Subject ที่สื่อความหมาย และใส่รายละเอียดปัญหาในเนื้อหา
3. แนบ Screenshot หรือไฟล์ที่เกี่ยวข้องได้
4. เมื่อระบบสร้าง Ticket แล้ว ให้ใช้ Ticket Number ที่ได้รับในการติดตาม
5. หากตอบกลับ Email เดิม ให้ใช้ปุ่ม Reply และอย่าลบข้อมูลอ้างอิงของ Email เดิม เพื่อให้ระบบจับเข้ากับ Ticket เดิม

> Email Alias จริงอาจแตกต่างกันตามบริษัท ให้ตรวจสอบจากผู้ดูแลระบบก่อนทำ UAT เช่น `helpdesk@บริษัท` หรือ Alias ที่ตั้งค่าไว้ใน Odoo

## 3. การติดตามและตอบกลับ Ticket

1. เปิด **My Helpdesk**
2. เลือก Ticket ที่ต้องการดู
3. ตรวจสอบ Subject, Priority, Status, รายละเอียด และประวัติ Conversation
4. หากต้องการส่งข้อมูลเพิ่ม ให้พิมพ์ข้อความในช่องตอบกลับ
5. แนบไฟล์เพิ่มเติมได้หลายรายการ
6. กด **Reply**
7. ดาวน์โหลดไฟล์เดิมจากหัวข้อ **Attachments** ได้

ควรตอบกลับใน Ticket เดิมเสมอ ไม่ควรสร้าง Ticket ใหม่สำหรับปัญหาเดียวกัน

## 4. การเลือก Priority

| Priority | ใช้เมื่อ | ตัวอย่าง |
|---|---|---|
| Low | ไม่เร่งด่วนและยังทำงานต่อได้ | ขอคำแนะนำทั่วไป |
| Medium | กระทบงานบางส่วน แต่มีวิธีทำงานชั่วคราว | โปรแกรมบางฟังก์ชันใช้งานไม่ได้ |
| High | กระทบผู้ใช้หรือกระบวนการสำคัญหลายคน | ระบบงานหลักทำงานผิดปกติ |
| Critical | งานหยุดชะงักหรือกระทบธุรกิจอย่างรุนแรง | ระบบหลักใช้งานไม่ได้ทั้งหน่วยงาน |

Priority ไม่ควรใช้แทนการระบุรายละเอียดปัญหา ควรแจ้งจำนวนผู้ได้รับผลกระทบและผลกระทบต่อธุรกิจด้วย

## 5. ความหมายของสถานะ Ticket

| สถานะ | ความหมาย |
|---|---|
| New | ระบบรับเรื่องแล้วและยังไม่ได้เริ่มดำเนินการ |
| Assigned | มีผู้รับผิดชอบแล้ว |
| In Progress | Agent กำลังตรวจสอบหรือแก้ไข |
| Pending User | รอข้อมูลหรือการยืนยันจากผู้แจ้ง สถานะนี้จะหยุดการนับ SLA |
| Resolved | แก้ไขปัญหาแล้ว รอการปิดงาน |
| Closed | ปิดงานเรียบร้อยแล้ว |
| Cancelled | ยกเลิก Ticket โดยไม่ดำเนินการต่อ |

การปิดงานต้องผ่านสถานะ **Resolved** ก่อน หากต้องแก้ไข Ticket ที่ปิดแล้ว Agent สามารถใช้ **Reopen** เพื่อกลับไปทำงานต่อ

## 6. ขั้นตอนการทำงานสำหรับ Agent

1. เปิด **IT Management > Helpdesk > Tickets**
2. ตรวจสอบรายการตาม Priority, Stage, Team และ SLA Deadline
3. เปิด Ticket ที่ยังไม่มีผู้รับผิดชอบ
4. กด **Assign to Me** หรือกำหนดผู้รับผิดชอบตามกระบวนการของทีม
5. ตรวจสอบรายละเอียด ผู้แจ้ง Category ไฟล์แนบ และประวัติ Chatter
6. เปลี่ยนสถานะเป็น **In Progress** เมื่อเริ่มดำเนินการ
7. ตอบกลับผู้แจ้งผ่าน Chatter หรือ Email ที่เชื่อมกับ Ticket
8. หากต้องการข้อมูลจากผู้แจ้ง เปลี่ยนเป็น **Pending User** และระบุคำถามให้ชัดเจน
9. เมื่อแก้ไขเสร็จ ให้บันทึกวิธีแก้ไขและเปลี่ยนเป็น **Resolved**
10. หลังยืนยันผลแล้วจึงกด **Close**

ปุ่มที่มีใน Ticket ได้แก่ **Assign to Me**, **Resolve**, **Close** และ **Reopen** การเปลี่ยนสถานะอื่นให้ใช้ Statusbar ตามสิทธิ์และขั้นตอนขององค์กร

## 7. SLA และการแจ้งเตือน

- SLA ใช้เวลาทำการตาม Resource Calendar ของบริษัท
- วันหยุดจะไม่ถูกนับเป็นเวลาทำการ
- เมื่อเป็น **Pending User** ระบบจะหยุดนับ SLA และเลื่อนกำหนดเวลาเมื่อกลับมาทำงานต่อ
- ตรวจสอบได้จากแท็บ **SLA** ได้แก่ Response Deadline, First Response, SLA Deadline และสถานะ Overdue
- หาก Ticket เกินกำหนด ระบบ Cron จะตรวจสอบและบันทึกการแจ้งเตือน SLA Overdue

เมื่อพบ SLA ใกล้ครบกำหนด ควรเร่งดำเนินการหรือแจ้ง Manager ก่อนถึงกำหนด ไม่ควรรอให้ระบบขึ้น Overdue

## 8. Knowledge Base สำหรับผู้ดูแล

1. เปิด **IT Management > Helpdesk > Knowledge Base**
2. สร้างบทความด้วยชื่อที่ค้นหาได้ง่าย
3. เลือก Category และเพิ่ม Tags ที่เกี่ยวข้อง
4. เขียนวิธีแก้ไขเป็นขั้นตอน พร้อมภาพหรือข้อควรระวัง
5. กด **Publish** เมื่อบทความพร้อมใช้งาน
6. ใช้ **Unpublish** เมื่อเนื้อหาไม่ถูกต้องหรืออยู่ระหว่างปรับปรุง
7. จากบทความสามารถกด **Create Ticket** เพื่อสร้าง Ticket ต่อได้

## 11. ข้อมูลที่ใช้ทดสอบแนะนำ

สร้างข้อมูลทดสอบอย่างน้อย:

- User Requester บริษัทที่ 1 จำนวน 1 คน
- Agent บริษัทที่ 1 จำนวน 1 คน
- Manager บริษัทที่ 1 จำนวน 1 คน
- User อีกบริษัทหนึ่งจำนวน 1 คน
- Ticket ระดับ Low, Medium, High และ Critical
- Ticket ที่มีและไม่มีไฟล์แนบ
- Ticket ที่มีผู้รับผิดชอบและ Unassigned
- Ticket ที่อยู่ในแต่ละ Stage
- SLA ที่มีเวลาทำการและวันหยุดบริษัท

## 13. เกณฑ์พร้อมใช้งาน

ถือว่าพร้อมสำหรับการเปิดใช้งานเมื่อผ่านเงื่อนไขต่อไปนี้:

- User สร้างและติดตาม Ticket ผ่าน Portal ได้
- Email ใหม่สร้าง Ticket ได้ และ Email ตอบกลับไม่สร้าง Ticket ซ้ำ
- User เห็นเฉพาะ Ticket ของตนเอง
- Agent รับงานและดำเนินการตามสถานะได้
- Pending User หยุด SLA และ Resume กลับมานับต่อได้
- Resolve และ Close ทำงานตามลำดับที่กำหนด
- Attachment เปิดดูและดาวน์โหลดได้ตามสิทธิ์
- Agent/Manager แยกข้อมูลตาม Company ได้ถูกต้อง
- Dashboard และรายงานตรวจสอบกับ Ticket จริงได้
- UAT ทุกกรณีระดับ Critical/High ผ่าน และไม่มี Defect ที่กระทบการใช้งานหลัก
